# 待處理項目與建議處理方式

紀錄目前已知但尚未處理的解析限制。每項附**成因、建議做法、風險、驗收方式**。
處理任何一項前，建議先叫用 `/rule-fix-loop`：先重現真實失敗案例 → 分類 → 動程式前先量化討論規則 → 拋棄式腳本驗證 → 最小改動 → 再驗證。

基準資料：`115年度預算案 (2).pdf`（共 704 列）。

---

## B. 一句列多科目＋合共（150 列，21%）〔優先度：中〕

**現象**
```
郵電費－編列3,225千元、電話費4,800千元及數據通信費3,386千元，合共11,411千元。
材料及用品費－編列辦公(事務)用品300千元及其他用品消耗750千元，合共1,050千元。
```
一列裡列了多個三級科目、各自帶金額，最後一個「合共」是小計。目前解析器**一列只給一個科目、一個金額**（金額取「合共」，取值是對的），所以個別三級科目的金額沒被拆出來。

**影響**
- 主表無法呈現各三級科目的金額。
- 金額一致性對帳的殘差主要來源（農村再生基金 `捐助、補助與獎助`、`專業服務費` 各約 10%）。

**建議做法**
1. 偵測條件：`description` 中可比對到科目表的「科目名＋N千元」配對 **≥2 個**，且句中有「合共N千元」。
2. 拆成多列子項，每個科目一列；原「合共」列標記 `isSubtotal: true`。
3. **安全閥（關鍵）**：只有在「子項金額加總 === 合共金額」時才拆；對不上就**維持原狀不拆**。這條讓改動自我驗證，不確定就不動。
4. 對帳彙總只計入 `!isSubtotal` 的列，避免小計與子項重複相加。

**風險**
主表列數、匯出內容、對帳彙總三處都會受影響；重複計算是主要風險，靠步驟 3、4 控制。

**驗收**
- 拆列前後，各基金／二級科目的金額總和**必須不變**。
- 對帳 tab 的「金額不一致」筆數應下降，且不得出現新的大額差異。

## C. 無科目名的用途活動句（182 列，26%）〔優先度：低／建議維持現狀〕

**現象**
```
提升飼料安全品質及強化風險管理等40,000千元。
建構韌性養豬產業及導入高效智慧生產模式等242,000千元。
農業技術團170,000千元，培訓農業技術人員作為農業師傅…
```
寫的是「做什麼」與金額，**原文完全沒有會計科目名**。這些多半出現在「一級科目標題句」之後，其加總等於該標題句的總額。

**建議做法：維持現狀（空白＋黃底標記待人工複核）**
理由：科目資訊不存在於文字中，任何規則都是臆測。已有的補救已經到位：
- 一級標題句裡的內嵌明細（`包括補(協)助政府機關(構)891,000千元、捐助國內團體511,000千元`）已被對帳抽取歸戶，金額層面不會漏。
- 這些列在主表以黃底 `.acct-filled` 標示，提醒人工確認。

**不建議的做法**
依標題句的內嵌比例回推分攤到各活動句——那是**捏造**分類，會讓錯誤看起來像正確答案。

## D. `parse_detail.py` 欄位偵測漏掉整個基金〔優先度：低〕

**現象**
`林務發展及造林基金` 的明細表頁（p172-174）被整批跳過，該基金 32 個二級科目在明細側完全消失。

**成因**
`_find_columns` 用**精確比對** `pos["前年度"]`，但該頁表頭是單一 cell「前年度決算數」（未被空白拆開）→ 找不到欄位錨點 → `if not cols: continue`。

**建議做法**
比照 JS 版 `_detailColumns` 的多形式比對：
```python
def px(*keys):
    for k in keys:
        if k in pos: return pos[k]
    return None
xF = px("前年度決算數", "前年度", "前")
```

**注意：影響範圍很小**
`parse_detail.py` **不被網頁工具使用**（明細版 tab 用的是 index.html 內的 JS），**也沒有掛進 MCP server**（`mcp_server.py` 只註冊 `parse_budget`）。僅在直接用 CLI 跑這支腳本時才有影響。

**驗收**
`parse_detail_pdf()` 應解析出 `林務發展及造林基金` 的二級科目（約 32 個）。

## E. 對帳 tab「僅一節出現」筆數偏多（約 86 筆）〔優先度：低〕

**成因（兩種，性質不同）**
1. **正常的粒度差異**：明細表逐項列出每個有金額的二級科目；編列說明只描述有文字說明者。「僅明細版有」多屬此類，**不是錯誤**。
2. **計畫命名對不齊**：說明版與明細版建立計畫階層的方式不同，同一科目落到不同計畫 key，兩邊都變成「僅一節有」。

**建議做法**
加一個「**基金層級／計畫層級**」切換鈕。基金層級會把計畫命名差異吸收掉，適合快速確認總量；計畫層級適合定位到具體計畫。目前「僅一節」區塊已預設摺疊，暫時夠用。
