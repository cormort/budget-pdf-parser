# 預算書 PDF 解析工具

把政府特種基金預算書 PDF，轉成可篩選、可匯出的結構化表格。

線上版：<https://cormort.github.io/budget-pdf-parser/>

---

## 一、要解決的問題

政府特種基金預算書是數百頁的 PDF，其中「**基金用途明細表說明**」章節以**自然語言段落**逐項敘述每筆預算的編列理由，例如：

```
壹、提升農業經營及發展計畫
一、糧政業務計畫
服務費用－編列16,982千元，包括：
1.國內旅費1,182千元。
2.肥料檢驗儀器機械及設備修護費500千元。
3.肥料品質檢驗分析專業人員勞務承攬6人之外包費3,500千元。
```

要拿它做分析，得先人工整理成「基金／計畫／一級科目／二級科目／三級科目／金額／編列說明」的表格。數百頁、上千列，純手工不可行。

困難點在於**這段文字沒有表格結構**：

- 階層靠中文標號（壹、一、（一）、1.）與縮排暗示，沒有欄位。
- 會計科目名可能在句首、句中、或只出現在上層標題句裡。
- 金額散落在敘述中，一句話可能有好幾個數字，還夾雜計算式。
- PDF 文字抽取會在意外的地方插入空白、把一行折成兩行。

同一份預算書還有另一種表述：「**基金用途明細表**」（純數字表格，科目對金額一對一）。兩者內容應該一致，但沒有現成工具能交叉檢查。

---

## 二、解決方案

單一 HTML 檔（`index.html`）的純前端工具，用 pdf.js 讀 PDF、SheetJS 匯出 xlsx，**不上傳任何檔案**。三個分頁：

### Tab 1｜用途明細表說明（編列說明）
上傳 PDF → 自動抓出「基金用途明細表說明」章節 → 清理文字 → 解析成表格 → 可依基金／計畫／科目篩選、匯出 xlsx。

### Tab 2｜用途明細表（數字版）
解析純數字的明細表，呈現 本年度預算／上年度預算／前年度決算 的比較；支援多檔並排。

### Tab 3｜金額一致性對帳（提醒工具）
上傳同一份 PDF，用兩個章節各自解析出的金額，在「基金＋計畫＋一級＋二級科目」層級比對，列出**金額不一致**或**僅一節出現**的項目，提醒人工複核。

另有 Python 版（`parse_budget.py` + `mcp_server.py`）提供相同的說明版解析能力，可掛成 MCP server 或 CLI 使用。

## 三、思考過程：七個核心難題

這份工具的價值不在程式碼量，而在**面對「沒有結構的結構化資料」時所做的一連串取捨**。

### 1. 文字沒有欄位 → 狀態機 ＋ 科目表比對

逐行掃描，維護一組「目前狀態」（基金／計畫三層／預算別／科目三層）。每行先判斷是**標題行**（更新狀態、不產列）還是**資料行**（產一列）。科目層級不靠縮排猜，而是拿**官方會計科目表**（1 用人費用 / 11 正式員額薪資 / 111 管理會委員報酬…）比對名稱，用科目代碼長度決定層級。

**為什麼**：縮排與標號在 PDF 抽取後常失真，科目表是唯一穩定的事實來源。

### 2. 科目名不總在句首 → 由精到鬆的回退鏈

實務上科目名出現的位置千奇百怪，於是排成一條**由精確到寬鬆**的比對鏈，前一層抓不到才輪到下一層：

| 順位 | 規則 | 例子 |
|---|---|---|
| 1 | 句首精確比對 | `國內旅費1,182千元` |
| 2 | 數字前綴後模糊比對 | `2.肥料檢驗儀器機械及設備修護費500千元` |
| 3 | 特徵字別名 | `…勞務承攬6人之外包費3,500千元` → 外包費 |
| 4 | 前一／下一科目關鍵字回退 | 依代碼順序的鄰近科目 |
| 5 | 補助對象推論 | `補助地方政府…` → 補(協)助政府機關(構) |
| 6 | 句中唯一科目名 | `…所需推展費30,403千元` |

**為什麼這個順序**：越後面的規則越可能誤判，所以讓它們**只看前面規則處理不了的殘餘**，把風險暴露面壓到最小。

### 3. 推論不能偽裝成事實 → 黃底標記

規則 3～6 得到的科目是**推論**，不是原文寫的。這些列在表格中以**黃底**標示，明說「這格是補的，請人工確認」。

**為什麼**：一個標記為待確認的推論值有用；一個看起來跟事實一樣的推論值是有害的。寧可空白加標記，也不要臆測後假裝確定。

### 4. 順序單調性給的是「界限」，不是「答案」

預算書的敘述**依科目代碼順序排列**（實測 397 組相鄰確定科目只有 4 組逆序，約 1%）。這條線索很強，但它能給的有限：

- 若某段孤兒句前後確定科目**相同**（A…A），中間必為 A → 可**移除黃底**，升級為確定。
- 若前後**不同**（A…C），只知道落在 [A, C] 區間，**無法指出是哪一個**。

所以只用它做兩件事：**A==C 時升級信心**、**推論結果超出 [A,C] 時收回**。刻意**不去猜**中間的 B。

**為什麼**：結構約束能排除錯誤答案，但排除不完的部分就是不知道。把「不知道」誠實標示出來，比填一個看起來合理的值好。

### 5. 清理階段解決問題，勝過在解析階段打補丁

PDF 換行會讓「編列100,963**千 元**」中間多一個空格，導致金額 regex（`千元`）抓不到，金額欄空白。

第一版的修法是把 regex 放寬成 `千\s*元`——治標。**真正的解法是讓清理階段真的移除行內空白**（原本只是把多個空白併成一個），讓解析器吃到的是乾淨文字。

**為什麼**：在管線上游把資料弄乾淨，下游所有規則都受益；在下游逐條放寬 regex，是打不完的地鼠。

### 6. 小計與明細不能一起加 → 逐區塊判斷

編列說明裡，同一個二級科目可能有**多個區塊**，有的寫了合計、有的只列明細：

```
捐助、補助與獎助－編列15,241千元，包括：   ← 合計
  1.補助地方政府…29,642千元                ← 明細
```

- 只取合計 → 沒寫合計的區塊整個漏掉（實測造成差異清一色「說明 < 明細」）。
- 全部加總 → 合計與明細重複相加（實測灌出假的 +29.9 億差額）。

正解是**逐區塊判斷**：該區塊寫了合計就用合計並跳過其明細；沒寫合計才加總其明細。**兩者不混用**。

**為什麼**：這個 bug 的線索是「差異清一色同方向」。單向偏誤幾乎必然是系統性錯誤，不是真實差異——**差異的分佈形狀，比差異的大小更能指出病灶**。

### 7. 驗證從哪裡來 → 讓另一份資料當基準

解析啟發式規則最難的是「怎麼知道對不對」。答案就在同一份 PDF 裡：**用途明細表的科目金額是一對一的表格，基本不會解析錯**。

於是 Tab 3 對帳工具實際上是**說明版解析器的 QA 工具**——任何金額不一致，優先假設是說明版解析有問題。這條回饋迴路在開發期間連續抓出四類 bug：

| 對帳版本 | 不一致 | 相符 | 抓到的成因 |
|---|---|---|---|
| 初版 | 21 | 96 | — |
| 第二版 | 23 | 99 | 「分述如下」小計重複加總 |
| 第三版 | 15 | 107 | 「有合計就只取合計」漏掉無合計區塊 |
| 第四版 | 16 | 108 | 二級金額只寫在一級標題句中 |
| 第五版 | ~12 | 更高 | 科目名接在「編列」之後 |

**為什麼**：沒有標準答案時，**找第二個資料來源互相驗證**，比盯著單一輸出反覆目視有效得多。

## 四、貫穿全案的原則

1. **寧可空白＋標記，不要臆測**：不確定的分類標成待複核，不填看似合理的值。
2. **先量化再改規則**：動程式前先跑真實資料，算出「會救到幾筆、會誤傷幾筆」。
3. **規則鏈由精到鬆**：風險高的規則放最後，只處理殘餘。
4. **界限優於猜測**：用結構約束排除錯誤答案，排除不掉的就承認不知道。
5. **在上游修，不在下游補**：資料清理階段解決，勝過逐條放寬比對規則。
6. **知道何時停**：規則加到後來只能多救幾筆卻開始引入錯誤時，就該停手，讓人工複核接手。

這套工作方法已抽成 `/rule-fix-loop` 技能（`~/.claude/skills/rule-fix-loop/`），適用於任何「改資料管線啟發式規則」的場景。
